# Contributing to Vervet

Thanks for your interest in contributing to Vervet! This guide will help you get set up and familiar with the project.

## Getting Started

### Prerequisites

- [Go](https://go.dev/) 1.24+
- [Bun](https://bun.sh/)
- [Wails CLI](https://wails.io/docs/gettingstarted/installation)

### Setup

```bash
git clone https://github.com/blacktau/vervet.git
cd vervet
wails dev
```

This starts the Go backend and Vite dev server together with hot-reload.

### Running Tests

```bash
# Go tests
go test ./...

# Frontend tests (from frontend/)
cd frontend
bun run test
```

### Linting

```bash
# From frontend/
bun run lint      # ESLint + type-check
bun run format    # Prettier
```

## Project Structure

- **Backend (Go):** `internal/` — organised by domain (`api/`, `servers/`, `connections/`, etc.)
- **Frontend (Vue 3):** `frontend/src/` — organised by feature (`features/`, `stores/`, `utils/`)
- **Wails bindings:** `frontend/wailsjs/` — auto-generated, do not edit manually

See [`CLAUDE.md`](CLAUDE.md) for a detailed architecture overview.

## Development Guidelines

### Code Style

- **Go:** follow standard Go conventions; `go vet` and `go fmt` should pass cleanly
- **TypeScript/Vue:** enforced by ESLint and Prettier — run `bun run lint` and `bun run format`
- No braceless `if` statements — always use curly braces
- Prefer early returns over nested `if`/`else`

### Do Not

- Edit files in `frontend/wailsjs/` — these are regenerated by Wails
- Run `bun run build` during development — it overwrites the frontend served by `wails dev`
- Store connection URIs in config files — they must go through the OS keyring

### Commits

- Write clear, concise commit messages
- Keep commits focused on a single change

## Submitting Changes

1. Fork the repository
2. Create a feature branch from `main`
3. Make your changes
4. Run tests and linting
5. Open a pull request against `main`

## Reporting Issues

Please use the [issue tracker](https://github.com/blacktau/vervet/issues) with the appropriate template:

- **Bug reports** — include steps to reproduce, expected vs actual behaviour, and your OS/platform
- **Feature requests** — describe the use case and what you'd like to see
